# 跨集高光切片（Cross-Episode Highlight Clip）设计

- 日期：2026-06-16
- 模块：`SimpleClaw/Flowcut/`
- 状态：设计已确认，待 writing-plans

## 1. 背景与目标

从一部剧的**原片全集**中，自动产出若干条**约 1 分钟的连续切片**用于投放：

- 从**前 3 集**里识别高光起点（高光可能在第 1、2 集，也可能直到第 3 集才出现）。
- 从该起点开始**连续往后播放原片**，**可跨集**拼接（一集若只有 20s，会自然拼到第 4、5、6 集）。
- 总时长目标 ~1 分钟，**不严格**：在 **50–70s** 窗口内的**逻辑边界**收尾，优先「一句话说完」，其次「分镜结束」，避免卡在画面/语义中间。
- 中间**不跳回原片开头**——是一段从起点起的连续切片。
- 一部剧产出**多条候选**（默认 3 条），各自异步合成，进入成片库供挑选。

### 与现有「单集高光」子系统的区别

现有 `extract_highlight_video` / `create_highlight_batch` → `scene_decompose` → `video_compose` 做的是「高光前置 teaser + 接回**同一集**整段原片」。本设计是**全新的连续跨集切片策略**，不替换现有逻辑，并行新增。

## 2. 总体架构与数据流

新增一条编排：`HIGHLIGHT_PLAN` 流 + 一个规划执行器。一部剧一个 durable 任务，服务端串起整条流水线，避免「任务等任务」的协调复杂度。

```
新工具 create_cross_episode_highlights(drama_name, num_candidates=3)
        │  按 drama_name 从 fc_highlight_asset 取 asset_type='episode_source' 全集，按 episode_no 排序
        ▼
[HIGHLIGHT_PLAN executor]   ← 全部编排在这一个执行器里
  Phase 1 起点检测（合并）:
    - 取前 3 集 → _normalize_clip 统一规格 → _ffmpeg_concat 拼成一条临时视频
    - ffprobe 记每集真实时长，建偏移表 offsets=[(ep1,0),(ep2,t1),(ep3,t2)]
    - 把拼接视频一次性上传 Gemini（highlight/base_prompt.md）
      → 返回跨集 segments（hook_strength/copy/全局时间戳）+ 高光候选
    - 取 top-N 高光候选；用偏移表把全局 start_time 反算成 (episode_no, local_start)
  Phase 2 前向扩展（逐集懒解，可越过第 3 集）:
    - 对每个起点候选，从起点集起向后逐集单集解（analyze_video，只要 copy + 分镜边界）
      拼成跨集时间线，累计到 50–70s 窗口；前 3 集 segment 复用 Phase 1 输出
    - 在窗口内选收尾边界（句子优先 → 分镜其次 → hard_cut 兜底）
    - 反算 clip_plan = [{asset_id, episode_no, cut_start, cut_end, oss_key}, ...]
  Phase 3 落库入队:
    - 每个候选建一条 fc_creative(creative_type='continuous_cross_episode',
      clip_plan_json=...), 入队 VIDEO_COMPOSE
        ▼
[VIDEO_COMPOSE executor]（扩展现有，不破坏单集逻辑）
  - 检测到 clip_plan_json → 逐集下载、裁切、标准化、concat → 上传 OSS → creative READY
  - 无 clip_plan_json → 走现有单集逻辑
        ▼
前端：复用 GET /tasks/{task_id} 轮询；多条 creative 异步出现在成片库 /creative
```

## 3. 算法模块 `Flowcut/services/clip_planner.py`（纯函数，可单测、无 IO）

IO（下载 / Gemini / ffmpeg / 落库 / 入队）全部留在执行器，算法层不碰。

### ① 起点选择（前 3 集合并后的 Gemini 输出）
- 输入：Phase 1 合并调用返回的高光候选（带 `hook_strength`/`candidate_use`/`context_dependency`/`continuity_risk`/全局 `start_time`）+ 偏移表。
- 排序键：`candidate_use ∈ {primary_hook, secondary_hook}` 优先 → `hook_strength` 高 → `context_dependency` 低 → `continuity_risk` 低。
- 去重：全局起点时间相近（<2s）的候选只留最高分，避免多条切片起点雷同。
- 取 top-N（默认 3）。每个候选用偏移表反算成 `StartCandidate(episode_no, local_start, global_start, reason)`。

### ② 跨集时间线（懒解）
- 从起点 segment 起，依次拼当前集剩余 segment → 下一集全部 segment → …，给每个 segment 标注全局累计时间 `cum`（起点 = 0）。
- 每解一集才下载 + 调一次 Gemini（前向单集解）；累计时长 ≥ 70s（上窗）即停止往后解，不浪费解整部剧。
- 前 3 集 segment 复用 Phase 1 合并输出（按偏移表切回各集），不重复解。

### ③ 收尾边界（窗口 50–70s，句子优先）
- 候选边界 = 落在 `cum ∈ [50, 70]` 区间内的每个 segment 的结束点。
- 「句子完整」标记：该 segment 的 `copy` 以句末标点结尾（`。！？…」"`），或下一 segment 的 copy 明显另起一句 / 换人说话。
- 选择规则：
  1. 句子完整的边界里，选 `cum` 最接近 60s 的；
  2. 没有句子完整的，退取窗口内最接近 60s 的分镜切点（segment 结束点即分镜切点）；
  3. 窗口内一个 segment 边界都没有（极端：超长镜头横跨整窗）→ 硬切 60s，标 `boundary_type='hard_cut'`。

### ④ clip_plan 反算（全局区间 → 各集裁切）
```
起点集:  [local_start, 该集末]
中间集:  [0, 该集末]            （整集）
收尾集:  [0, end_local]
```
输出 `ClipPlan`：`entries=[{asset_id, episode_no, cut_start, cut_end, oss_key}, ...]` + 元数据（`total_duration`、`boundary_type`、`start_reason`）。

### 纯函数签名
```python
select_start_candidates(highlight_candidates, offsets, top_n) -> list[StartCandidate]
build_timeline(start, forward_segments) -> list[TimedSegment]   # 带 cum
pick_end_boundary(timeline, window=(50,70), ideal=60) -> EndBoundary
build_clip_plan(start, end_boundary, episodes) -> ClipPlan
```

- `fc_creative` 新增 `clip_plan_json JSON NULL`，`ensure_schema` 里 `ADD COLUMN IF NOT EXISTS`，向后兼容。
- 新 `creative_type='continuous_cross_episode'`；`drama_name` 复用现有列；`script_id` = 起点集 fc_script（仅溯源，compose 不靠它）。
- `highlight_start/end` 存「起点集内起止」，仅供前端展示；真正拼接看 `clip_plan_json`。
- `CreativeRepository`：新增 `create_cross_episode_job(...)`；`_row_to_dict` 解析 `clip_plan_json`。

## 5. VIDEO_COMPOSE 执行器扩展（不破坏现有）

```python
if creative.get("clip_plan_json"):        # 新分支
    for entry in clip_plan.entries:       # 严格按 episode_no 顺序
        下载该集 → _ffmpeg_cut_clip(cut_start, cut_end) → _normalize_clip
    _ffmpeg_concat(所有裁片) → 上传 OSS → creative READY

```
- 分辨率 / 帧率不一时 `_normalize_clip` 已统一。
- 输出 OSS key 沿用 `creatives/{tenant_key}/highlight/{creative_id}/{uuid}.mp4`。

## 6. 工具 / API 出入参

- 新工具 `create_cross_episode_highlights`：
  - 入参：`drama_name`(必填)、`num_candidates`(默认 3)。
  - 时长窗口 `[50,70]` 在 MVP 为固定常量（`clip_planner.WINDOW`），不开放为工具入参；后续如需可加 `target_window` 透传给 `pick_end_boundary`。
  - 行为：建一条 `HIGHLIGHT_PLAN` durable 任务，立即 ack，返回 `task_id` + navigate 到 `/creative`。
- 轮询复用现有 `GET /tasks/{task_id}`；规划完成后多条 creative 出现在成片库各自异步 compose。
- 常量集中放执行器 / config：`START_SEARCH_EPISODES=3`、`WINDOW=(50,70)`、`IDEAL=60`、`DEFAULT_CANDIDATES=3`、`MAX_FORWARD_EPISODES=6`（兜底防呆）。

## 7. 错误处理（显式、不吞）

- 全集 < 1 集 / 前 3 集合并解失败 → 任务 FAILED，带可读原因。
- 某候选向后集数不足以凑到 50s（剧太短）→ 跳过该候选并记录，不让整任务挂。
- 收尾落入 `hard_cut` 兜底 → creative 标 `boundary_type='hard_cut'`，前端提示「未找到理想边界」。
- compose 阶段任一集下载 / 裁切失败 → 该 creative FAILED，其余候选不受影响。
- Phase 1 临时拼接视频、Phase 2 临时裁片，`finally` 清理临时目录。

## 9. 关键决策记录

- 起点检测用**前 3 集合并 + 一次 Gemini 调用**（跨集连续画面里整体判断高光），不逐集排序。
- 前向扩展用**逐集单集解**（只需 copy + 分镜边界，省 token），可越过第 3 集。
- 收尾时长窗口 **50–70s**，**句子说完优先**于分镜切点。
- 一部剧产出**多条候选**（默认 3），各自异步合成。
- 规划全部在一个服务端 durable 执行器内同步完成，不引入子任务等待。
