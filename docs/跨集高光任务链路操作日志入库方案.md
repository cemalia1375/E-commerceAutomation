# 跨集高光任务链路操作日志入库方案

日期：2026-07-11

## 1. 背景与目标

当前 FlowCut 客户端 exe 默认会连接云端 MySQL，跨集高光链路已有业务状态表：

- `fc_highlight_asset`：高光资产库，记录原片、数字人承接视频、前贴等资产。
- `fc_highlight_batch`：跨集高光批量任务主表，记录一个剧名维度的批量编排状态。
- `fc_highlight_stage`：跨集高光任务阶段表，记录单集预处理、合并拆解、起点选择、候选规划等阶段。
- `fc_creative`：成片表，已有 `creative_type/batch_id/source_asset_id/connector_asset_id/highlight_start/highlight_end` 等高光字段。
- `nb_runtime_tasks`：异步任务表，记录 worker 任务状态、payload、错误与结果。

这些表能反映业务结果，但不能作为“用户操作日志”。原因是：

- 业务表会被更新覆盖，不能完整保留用户点击、提交、取消、重试等行为历史。
- 部分前端行为失败时不会产生业务记录，例如用户选择文件后上传失败、接口 4xx/5xx、轮询超时。
- 排查时，需要按用户、租户、剧名、批次、阶段、按钮动作、错误链路进行检索。

需要新增独立操作日志表，作为跨集高光任务链路的审计与排障基础。

## 2. 范围边界

本方案仅覆盖跨集高光任务链路：

- 高光原片资产上传、zip 批量导入、删除。
- 跨集高光批量任务创建、取消、重试、查询。
- 高光链路阶段流转：单集预处理、合并拆解、起点选择、候选规划、成片合成。
- 高光成片查看、重新合成、导出下载。
- 链路内 API 请求失败、前端轮询超时、后端 worker 失败。

本方案暂不覆盖：

- 千川/巨量广告投放发布。
- 普通素材库、脚本生成、脚本匹配、素材脚本自动化。
- LLM token 级日志或完整 prompt 内容。

## 3. 表结构设计

两层表设计：

1. `fc_highlight_audit_event`：不可变操作日志表，记录用户/系统在跨集高光链路中的每一次关键操作。
2. `fc_highlight_pipeline_event`：可选的链路阶段事件表，记录批次和阶段的状态迁移，便于还原任务执行时间线。

## 5. 新增表一：`fc_highlight_audit_event`

定位：跨集高光用户操作日志与接口审计表。该表只追加，不更新，不删除，作为排障事实源。

### 5.2 字段说明

| 字段 | 说明                                                        |
|---|-----------------------------------------------------------|
| `event_id` | 前端或后端生成的唯一事件 ID，建议 UUID/hex，支持幂等去重                        |
| `tenant_key` | 用户标识，必填                                                   |
| `user_id/username` | 登录用户信息，由后端从 cookie/session 补齐                             |
| `session_key` | Agent 会话或页面会话标识；纯 API 操作可为空                               |
| `event_type` | 事件类型，例如 `highlight.batch.create.submit`                   |
| `event_source` | `client/api/worker/system`                                |
| `event_level` | `debug/info/warn/error`                                   |
| `status` | `ok/failed/started/submitted/succeeded/cancelled/timeout` |
| `page/route/component/action` | 前端页面与组件定位                                                 |
| `batch_id/stage_id/stage/runtime_task_id` | 跨集高光批次、阶段、异步任务关联                                          |
| `creative_id/asset_id` | 成片或资产关联                                                   |
| `drama_name/episode_no/candidate_idx` | 业务维度，便于运营排查                                               |
| `request_id` | 单次 API 请求 ID，前端 interceptor 和后端日志可共用                      |
| `http_method/api_path/http_status/duration_ms` | 接口维度统计                                                    |
| `external_provider/external_base_host/external_model/external_operation/external_http_status` | 外部模型/API 调用摘要，例如 Gemini 经 `moyu.info` 中转时的主机、模型、操作和 HTTP 状态 |
| `error_code/error_message` | 错误摘要，不存堆栈全文和敏感信息                                          |
| `payload_json` | 扩展字段，只放非敏感业务摘要                                            |
| `occurred_at` | 客户端或服务端事件发生时间                                             |
| `created_at` | 数据库入库时间                                                   |

## 7. 外部模型/API 调用错误记录规范

跨集高光链路中，起点选择、视频理解、候选规划等步骤会调用外部模型/API。当前生产包存在 `GEMINI_BASE_URL=https://moyu.info` 这类 Gemini 中转配置，操作日志必须明确覆盖 moyu/Gemini 中转失败。

### 7.1 记录目标

当后端 worker 或 API 调用外部模型/API 失败时，需要能回答：

- 是哪个租户、哪个用户、哪个剧、哪个批次触发的。
- 失败发生在哪个阶段：合并拆解、起点选择、候选规划等。
- 调用的是哪个外部提供方、哪个中转 host、哪个模型。
- HTTP 状态、错误类型、是否可重试、重试第几次。
- 失败后业务状态如何变化：阶段失败、重试、批次失败或部分成功。

### 7.2 写入字段

外部 API 错误优先写入结构化列：

```text
external_provider      例如 gemini / volcengine / embedding
external_base_host     例如 moyu.info
external_model         例如 gemini-3.5-flash
external_operation     例如 highlight_start_select / merge_decompose / span_plan
external_http_status   例如 429 / 500 / 502 / 504
error_code             例如 external_timeout / bad_gateway / response_invalid
error_message          截断后的错误摘要
```

更多细节写入 `payload_json` 或 `details_json`：

```json
{
  "external_provider": "gemini",
  "external_base_host": "moyu.info",
  "external_model": "gemini-3.5-flash",
  "external_operation": "highlight_start_select",
  "external_http_status": 502,
  "external_error_type": "bad_gateway",
  "retryable": true,
  "attempt": 1,
  "timeout_ms": 120000
}
```

### 7.3 事件类型

```text
highlight.external_api.request.failed
highlight.external_api.request.timeout
highlight.external_api.response_invalid
highlight.external_api.retry
pipeline.external_api.failed
pipeline.external_api.timeout
pipeline.external_api.response_invalid
```

### 7.4 安全要求

外部 API 错误日志禁止写入：

- API key、Authorization header、Cookie、代理认证信息。
- 完整 prompt、完整模型响应、完整视频内容。
- 用户本地文件绝对路径。

允许写入：

- host，例如 `moyu.info`。
- provider，例如 `gemini`。
- model 名称。
- HTTP 状态码。
- 截断后的错误摘要。
- 是否重试、重试次数、耗时。

## 8. 写入时机设计

### 8.1 前端写入

通过新增 `POST /highlight-audit/events` 接口上报，不阻塞主流程。失败时仅写 console，不弹窗。

前端埋点位置：

| 页面/组件 | 事件 |
|---|---|
| 高光资产上传弹窗 | 选择文件、提交上传、上传成功、上传失败 |
| 高光资产库 | 查看资产分组、删除单个资产、批量删除 |
| Chat 工具触发跨集高光 | 工具结果返回、跳转高光成片库 |
| 高光成片库 | 打开页面、筛选、重新合成、预览、导出、下载 |
| 全局 axios interceptor | `/highlight-*`、`/creatives/*highlight*` 请求失败与超时 |

### 8.2 后端 API 写入

API 层用于记录用户明确提交的动作结果：

| 接口 | 写入事件 |
|---|---|
| `POST /highlight-assets/upload` | `highlight.asset.upload.submit/success/failed` |
| `POST /highlight-assets/upload-zip` | `highlight.asset.zip_upload.submit/success/failed` |
| `DELETE /highlight-assets/{asset_id}` | `highlight.asset.delete.*` |
| `POST /highlight-assets/batch-delete` | `highlight.asset.batch_delete.*` |
| `POST /highlight-batches` | `highlight.batch.create.*` |
| `POST /highlight-batches/{batch_id}/cancel` | `highlight.batch.cancel.*` |
| `POST /highlight-batches/{batch_id}/retry` | `highlight.batch.retry.*` |
| `POST /creatives/{creative_id}/compose-highlight` | `highlight.creative.compose.*` |
| `POST /creatives/{creative_id}/export-highlight` | `highlight.creative.export.*` |

### 8.3 Worker 写入

Worker 层用于记录内部阶段：

| 模块 | 写入事件 |
|---|---|
| `highlight_batch.py` | batch 状态迁移、阶段创建、子任务提交 |
| `highlight_episode_prepare.py` | 单集下载/归一化开始、成功、失败 |
| `highlight_merge_decompose.py` | 合并、粗拆镜、merged shots 写入 |
| `highlight_start_select.py` | 起点选择开始、候选创建、重试、失败 |
| `highlight_span_plan.py` | 候选细拆、clip plan、creative 创建、compose 任务提交 |
| `executors.py` 中 highlight compose/export | 成片合成/导出成功失败 |

外部模型/API 调用失败建议在捕获异常处同步写入：

| 模块 | 外部调用错误事件 |
|---|---|
| `highlight_merge_decompose.py` | `pipeline.external_api.failed`，`external_operation=merge_decompose` |
| `highlight_start_select.py` | `pipeline.external_api.failed`，`external_operation=highlight_start_select` |
| `highlight_span_plan.py` | `pipeline.external_api.failed`，`external_operation=span_plan` |
| `gemini_video.py` | `highlight.external_api.request.failed/timeout/response_invalid` |

## 10. 安全与合规约束

日志中禁止写入：

- 密码、cookie、session token、API key、OSS access key。
- 完整视频内容、完整脚本文案、完整 LLM prompt。
- 超长异常堆栈。

做法：

- `error_message` 截断到 512 字符。
- `payload_json` 仅记录业务摘要和 ID。
- 前端文件上传日志只记录文件名、大小、类型、数量，不记录本地绝对路径。
- 后端统一对 `payload_json` 做脱敏过滤。

## 11. 查询示例

### 11.1 查询某个批次完整操作时间线

```sql
SELECT occurred_at, event_source, event_type, status, stage, runtime_task_id,
       creative_id, asset_id, error_message
FROM fc_highlight_audit_event
WHERE tenant_key = 'flowcut'
  AND batch_id = '目标_batch_id'
```

### 11.2 查询某个剧最近的失败

```sql
SELECT occurred_at, username, event_type, status, batch_id, stage,
       http_status, error_code, error_message
FROM fc_highlight_audit_event
WHERE tenant_key = 'flowcut'
  AND drama_name = '目标剧名'
  AND status IN ('failed', 'timeout')
ORDER BY occurred_at DESC
LIMIT 100;
```

### 11.3 查询上传成功但后续没有任务的剧

```sql
SELECT a.drama_name, COUNT(*) AS uploaded_assets, MAX(a.created_at) AS last_upload_at
FROM fc_highlight_asset a
LEFT JOIN fc_highlight_batch b
  ON b.tenant_key = a.tenant_key AND b.drama_name = a.drama_name
WHERE a.tenant_key = 'flowcut'
  AND a.asset_type = 'episode_source'
GROUP BY a.drama_name
HAVING COUNT(b.id) = 0
ORDER BY last_upload_at DESC;
```

### 11.4 查询阶段耗时

```sql
SELECT stage,
       COUNT(*) AS total,
       SUM(status = 'READY') AS ready_count,
       SUM(status = 'FAILED') AS failed_count,
       AVG(TIMESTAMPDIFF(SECOND, started_at, completed_at)) AS avg_seconds
FROM fc_highlight_stage
WHERE batch_id = '目标_batch_id'
GROUP BY stage;
```

### 11.5 查询用户最近高光操作

```sql
SELECT occurred_at, event_type, status, drama_name, batch_id, creative_id, error_message
FROM fc_highlight_audit_event
WHERE tenant_key = 'flowcut'
  AND username = '目标用户名'

LIMIT 100;
```

### 11.6 查询 moyu/Gemini 中转失败

```sql
SELECT occurred_at, tenant_key, username, drama_name, batch_id, stage,
       runtime_task_id, external_provider, external_base_host, external_model,
       external_operation, external_http_status, error_code, error_message
FROM fc_highlight_audit_event
WHERE tenant_key = 'flowcut'
  AND external_base_host = 'moyu.info'

```

### 11.7 统计外部 API 失败类型

```sql
SELECT external_base_host,
       external_model,
       external_operation,
       COUNT(*) AS total_errors,
       SUM(external_http_status = 429) AS rate_limited,
       SUM(external_http_status BETWEEN 500 AND 599) AS upstream_5xx,
       SUM(status = 'timeout') AS timeout_count
FROM fc_highlight_audit_event
WHERE tenant_key = 'flowcut'
  AND external_provider IS NOT NULL
  AND occurred_at >= DATE_SUB(NOW(), INTERVAL 7 DAY)
GROUP BY external_base_host, external_model, external_operation
ORDER BY total_errors DESC;
```

## 12. Agent 工具调用追踪补齐

跨集高光链路除了“用户操作日志”和“worker 阶段事件”外，还需要补齐 Agent 工具调用追踪。否则只能看到用户发起了任务、任务进入了 worker，但无法稳定回答：

- Agent 是否真的调用了 `create_cross_episode_highlights`。
- LLM 本轮还调用了哪些辅助工具。
- 工具入参是什么，返回结果是什么摘要。
- 工具调用是否成功、失败、被拦截、去重或提交成异步任务。
- 工具调用和 `batch_id/runtime_task_id` 如何关联。

当前 `nb_agent_tool_invocations` 已经会建表，但 FlowCut 的 `MainAgent` 还没有把 `invocation_store` 传入 `ToolRegistry/ToolExecutionRuntime`，所以这张表不能作为完整事实源。Mojing 已经接入该链路，FlowCut 按同样方式补齐。

### 12.1 复用表：`nb_agent_tool_invocations`

该表用于记录模型触发的工具调用，不另建 FlowCut 专属工具表。复用现有表可以保持 simpleclaw harness 层统一。

核心字段：

```sql
invocation_id      VARCHAR(64)  NOT NULL
tenant_key         VARCHAR(255) NOT NULL
session_key        VARCHAR(255) NOT NULL
message_seq        INT          NULL
tool_call_id       VARCHAR(128) NULL
tool_name          VARCHAR(128) NOT NULL
tool_category      VARCHAR(32)  NOT NULL DEFAULT 'sync_read'
execution_mode     VARCHAR(32)  NOT NULL DEFAULT 'immediate'
status             VARCHAR(32)  NOT NULL DEFAULT 'requested'
input_json         JSON         NULL
output_summary     TEXT         NULL
runtime_task_id    VARCHAR(64)  NULL
business_ref_type  VARCHAR(64)  NULL
business_ref_id    VARCHAR(128) NULL
trace_id           VARCHAR(64)  NULL
last_error         TEXT         NULL
created_at         DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP
updated_at         DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
completed_at       DATETIME     NULL
```

跨集高光写入约定：

| 字段 | 跨集高光写入建议 |
|---|---|
| `tool_name` | `create_cross_episode_highlights`、`check_task_status` 等 |
| `tool_category` | 创建高光任务用 `async_task`；查询类用 `sync_read` |
| `execution_mode` | durable 工具写 `durable` |
| `status` | `running/submitted/succeeded/failed/blocked/deduped` |
| `input_json` | 工具入参摘要，例如 `drama_name/num_candidates/connector_asset_id` |
| `output_summary` | 工具返回摘要，最多保留截断文本 |
| `runtime_task_id` | 关联 `nb_runtime_tasks.task_id` |
| `business_ref_type` | 写 `highlight_batch` |
| `business_ref_id` | 写 `fc_highlight_batch.batch_id` |

### 12.2 后端接入改造点

1. 新增 `Flowcut.storage.tool_invocation_repo.ToolInvocationRepository`。
   - 可复用 `Mojing.storage.tool_invocation_repo.ToolInvocationRepository` 实现。
   - 仅将 `Database` import 改为 `Flowcut.storage.database.Database`。

2. 在 `Flowcut.api.container.AppContainer` 增加字段：

```python
tool_invocation_repo: ToolInvocationRepository
```

3. 在 `Flowcut.api.container.build_container()` 初始化：

```python
tool_invocation_repo = ToolInvocationRepository(db)
```

4. 构造 `MainAgent` 时传入：

```python
main_agent = MainAgent(
    db=db,
    task_repo=task_repo,
    material_repo=material_repo,
    script_repo=script_repo,
    creative_repo=creative_repo,
    base_registry=base_registry,
    tool_factories=tool_factories,
    tool_invocation_store=tool_invocation_repo,
)
```

5. `Flowcut.agent.main_agent.MainAgent.__init__` 增加参数并保存：

```python
tool_invocation_store: ToolInvocationStore | None = None
self._tool_invocation_store = tool_invocation_store
```

6. `make_tool_registry()` 创建 `ToolRegistry` 时传入：

```python
reg = ToolRegistry(
    runtime_services=self._base_registry.runtime_services,
    tool_lifecycle=self._make_tool_lifecycle(),
    invocation_store=self._tool_invocation_store,
)
```

### 12.3 和高光审计表的关系

`nb_agent_tool_invocations` 与 `fc_highlight_audit_event/fc_highlight_pipeline_event` 不是替代关系，而是互补关系：

| 表 | 负责回答的问题 |
|---|---|
| `fc_highlight_audit_event` | 用户点了什么、API 是否成功、moyu/Gemini 是否报错 |
| `fc_highlight_pipeline_event` | 批次和阶段如何流转、worker 哪里失败 |
| `nb_agent_tool_invocations` | Agent/LLM 实际调用了哪个工具、入参和工具返回是什么 |
| `nb_runtime_tasks` | 异步任务是否入队、运行、成功或失败 |

跨表关联：

```text
nb_agent_tool_invocations.runtime_task_id -> nb_runtime_tasks.task_id
nb_agent_tool_invocations.business_ref_id -> fc_highlight_batch.batch_id
fc_highlight_audit_event.batch_id -> fc_highlight_batch.batch_id
fc_highlight_pipeline_event.runtime_task_id -> nb_runtime_tasks.task_id
```

## 14. 实际设计选项

### 选项 A：轻量版

只新增 `fc_highlight_audit_event`，所有用户操作、API 结果、worker 阶段事件都写入一张表。

优点： 表少，落地快。 查询简单，适合 MVP。

缺点： 操作日志和状态机事件混在一起，后期数据量大时需要更严格的事件规范。

### 选项 B：完整版

新增 `fc_highlight_audit_event` + `fc_highlight_pipeline_event`。

优点： 用户行为和任务阶段分层清晰。 适合后续做任务链路看板、失败率统计、阶段耗时分析。

缺点： 后端写入点更多，开发和测试成本略高。

A：为了上线后可查用户操作；
B：云数据库本次要一次性定型。
