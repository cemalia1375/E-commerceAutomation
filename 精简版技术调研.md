# AI 营销自动化 — 技术调研（精简版）

> MVP Phase 1 范围：爆款拆解 → 素材管理 → 脚本匹配 → AI补位 → 视频组合 → 千川上架 → 数据报表

先明确一点：剪辑用的"素材"怎么定义？是一段视频，还是说一段 BGM、一段文字、一个音频、一个视频都可以算？暂时按视频片段处理，其他类型等运营确认后再扩。

---

## 全局技术决策

| # | 决策项 | 决定 | 备注 |
|---|---|---|---|
| 1 | 字幕 | 素材原声 ASR 为唯一来源 | 入库时跑 ASR 写入 `material.transcript`；素材无口播则不带字幕；AI 补位段字幕规则见「待确认」 |
| 2 | BGM | MVP 不做 | 运营后期在剪映加 |
| 3 | AI 水印 | 不本地烧录 | 依赖生成模型自带水印；**POC 阶段必须验证** |
| 4 | 模块 3 输出 | 初剪 mp4 + SRT | 供运营导入剪映二次精修，不追求直接可投 |
| 5 | 千川上架 | Marketing API（企业开发者路径） | 自营场景，不走服务商；RPA 不做兜底 |
| 6 | 素材匹配 | 画面主匹配 + 口播二排 | 模块 2 同时存 `name`（画面）与 `transcript`（口播），分两阶段匹配；具体打分方案待导师方案对齐 |
| 7 | 主库 | PostgreSQL 单库 | 见模块 2 选型说明 |

## 第一部分：产品逻辑

### 整体流程

```
输入爆款视频（本地上传，30-40s）
        ↓
  多模态LLM拆解分镜脚本
  （JSON格式，每段含时间戳+画面描述+口播台词）
        ↓
  SceneDetect 时间边界修正
        ↓
  生成 3-5 条差异化脚本
        ↓
  用户上传素材 → 自动命名入库
  （类别-时长-内容 格式）
        ↓
  脚本匹配：按每段 content + 时长 搜索素材库
  输出：已匹配 / 低匹配 / 缺失 三档结果
        ↓
  Agent 循环检视已剪辑视频流畅度
  → 流畅：进入成片确认
  → 不流畅：定位问题段 → 搜索素材 or AI补位生成
        ↓
  成片预览 → 人工确认
        ↓
  一键上架千川（上传素材 + 生成标题）
        ↓
  投放数据回流 → 效果看板
```

### 模块 1：爆款拆解与脚本生成

多模态模型直接理解视频，一次调用同时完成画面理解和语音识别，输出每段的时间戳、画面描述、口播台词。时间边界用 PySceneDetect 做二次修正，把 LLM 的语义估算吸附到真实切镜点。最后基于拆解结果，生成 3-5 条差异化脚本（换钩子角度、痛点场景、产品呈现方式等）。

**模型方案**（均已核验 2026-05 公开可用）：
- 海外：**Gemini 3.1 Pro**（2026-02 GA，1M context，原生视频流理解，单次可处理 1 小时视频，非帧抽样）
- 国内主选：**Qwen3.5-Omni**（百炼，2026-03 发布，音视频基准 SOTA，原生多模态）
- 国内备选：**Kimi K2.5 / K2.6**（月之暗面，2026-01 起开源原生多模态，支持图像/视频流统一 token 编码）

POC 阶段需横向对比拆解精度、单次成本和延迟（见 POC 清单 P0）。

**成本基线**：Gemini 3.1 Pro 视频输入约 **100 tokens/s**（实测）。30-40s 视频单次拆解输入 ≈ 3-4K tokens，加上 prompt overhead 和输出 reasoning，单次调用预估 5-10K tokens，单条成本量级在分以内。模块 3 评估 Agent 同此口径。

### 模块 2：素材管理与脚本匹配

**素材入库**。上传视频后自动生成结构化命名：`类别-时长-内容描述`（如 `人物素材-3s-主播打开瓶口介绍产品`），同时跑 ASR 写入 `transcript` 字段（素材原声字幕的唯一来源）。异步生成缩略图（第 1 秒抽帧）和 5 秒预览片段。

**脚本匹配**。匹配维度为画面主匹配 + 口播二排：先按画面描述 (`name`) 与时长筛候选，再用口播 (`transcript`) 与脚本台词做二次排序。输出三档：
- 已匹配：画面与口播均命中，素材预览 + 匹配度分数
- 低匹配：仅画面命中或口播命中度低
- 缺失：库中无可用内容

用户查看后可手动替换或确认。具体打分公式与权重待导师方案对齐（见「待确认」）。

**选型说明（PostgreSQL 单库）**：
- 留出语义召回的退路：素材匹配只要沾 embedding，pgvector 是事实标准；MySQL 8.0.32+ 的 VECTOR 生态明显落后，后期升级要迁库。MVP 即使先不接 embedding，也用 PG 把这条路留好。

**表结构**：

```sql
CREATE TABLE material (
  id            BIGSERIAL      PRIMARY KEY,
  oss_key       VARCHAR(512)   NOT NULL,
  oss_url       VARCHAR(1024)  NOT NULL,
  thumbnail_url VARCHAR(1024),
  preview_url   VARCHAR(1024),
  name          VARCHAR(255)   NOT NULL,          -- 画面描述
  transcript    TEXT,                              -- 素材 ASR 原声台词；无口播为 NULL
  category      VARCHAR(32)    NOT NULL,           -- 人物/产品/场景/字幕板/氛围
  duration      REAL           NOT NULL,
  file_size     BIGINT         NOT NULL,
  status        VARCHAR(16)    DEFAULT 'PROCESSING', -- PROCESSING / READY / FAILED
  usage_count   INT            DEFAULT 0,
  created_at    TIMESTAMPTZ    DEFAULT NOW(),
  updated_at    TIMESTAMPTZ    DEFAULT NOW()
);

-- 中文全文检索（zhparser）
CREATE INDEX idx_material_name_fts
  ON material USING GIN (to_tsvector('zhcfg', name));
CREATE INDEX idx_material_transcript_fts
  ON material USING GIN (to_tsvector('zhcfg', coalesce(transcript, '')));
```

**OSS 路径**：`materials/{uuid}/` 存原始文件、缩略图、预览；`creatives/{uuid}/` 存成片。

### 模块 3：AI 补位与视频组合

两个子 Agent 独立运行：评估 Agent 看初剪序列判断流畅度（过渡突兀 / 画面缺失 / 时长异常），生成 Agent 调 **Seedance 2.0**（字节豆包，2026-02，**原生首尾帧模式 + 输出带音效/BGM**，单次最长 15s）做补位。

**Seedance 接入要点（已核实 2026-05）**：
- 走火山方舟 API；**企业公测需资格申请 + 保证金**，需提前启动
- 计费：纯生成 46 元/M tokens（≈ 1 元/秒）；含视频输入 28 元/M tokens
- 输出：480P / 720P 两档；并发数 10
- 成本测算：若日产 50 条成片、平均每条补 5-10s → **日均 250-500 元，月 7.5K-15K**
- 首尾帧具体调用方式需查火山方舟最新 SDK 文档

**流程**：初剪拼接 → 评估 Agent 输出问题段 → 主 Agent 先去素材库再搜一轮 → 没有合适的就触发 Seedance 首尾帧补全（前段尾帧 + 后段首帧为参考） → 更新序列 → 再次评估 → 循环到通过或达迭代上限（MVP 默认 3 次），转人工确认。

设置迭代上限而非"一开始就转人工"，是为后续走向全流程自动化（去人工卡点）保留迭代闭环。

由于 Seedance 输出含音频，AI 补位段可走主流程 ASR 提字幕，与素材一致；不再有"纯画面无字幕"问题。

**评估 Agent 输入契约**：每次调用同时传初剪 mp4 + 结构元数据，让 Gemini 知道分段边界，避免输出的 `segment_idx` 跟脚本分段对不上。

```json
{
  "video_url": "oss://.../draft_xxx.mp4",
  "total_duration": 35.0,
  "segments": [
    {"idx": 1, "start_sec": 0.0,  "end_sec": 3.2,  "material_id": "m_8821", "source": "library"},
    {"idx": 2, "start_sec": 3.2,  "end_sec": 8.5,  "material_id": "m_8829", "source": "library"},
    {"idx": 3, "start_sec": 8.5,  "end_sec": 14.0, "material_id": "sd_001", "source": "seedance"},
    ...
  ]
}
```

Prompt 中要求 Gemini 输出的 `segment_idx` 必须对齐此处的 `idx`，主 Agent 据此反查素材 id 触发对应修复。

**评估 Agent 输出 schema（初版）**：

```json
{
  "iteration": 1,
  "verdict": "PASS | FAIL",
  "overall_score": 0.0,
  "issues": [
    {
      "segment_idx": 2,
      "start_sec": 6.4,
      "end_sec": 8.2,
      "issue_type": "TRANSITION_ABRUPT | VISUAL_GAP | DURATION_ANOMALY | AUDIO_DISCONTINUITY | VISUAL_INCOHERENCE",
      "severity": "HIGH | MEDIUM | LOW",
      "description": "前段室内白光镜头跳到室外特写，色温差异大",
      "suggested_action": "SEARCH_LIBRARY | SEEDANCE_FILL | ADJUST_DURATION | REPLACE_MATERIAL",
      "fix_hint": "在 6.4s-7.4s 插入过渡画面，参考前段尾帧色温"
    }
  ]
}
```

字段说明：
- `verdict`：整体通过/不通过；`PASS` 即进入人工确认
- `segment_idx` / `start_sec` / `end_sec`：定位到具体段，让生成 Agent 能精准触发修复
- `issue_type`：枚举驱动后续动作分支，避免自由文本难解析
- `suggested_action`：直接告诉主 Agent 调度方向（搜库 / Seedance / 改时长 / 换素材）
- 流畅度判定的具体标准由运营撰写（纳入「待确认」），通过 prompt 注入此 schema 之前的判定规则段

MVP 输出两个文件：初剪 mp4 + SRT 字幕文件。字幕规则按决策 1：以素材 `transcript` 为唯一来源，按素材实际时间轴写入 SRT，素材无口播段不出字幕。

### 模块 4：千川上架

千川 Marketing API 覆盖素材上传、计划创建、数据回流全链路。

**鉴权**（已核实 2026-05）：
- access_token 24h / refresh_token 30 天
- access_token 过期可用 refresh_token 自动换取新 access_token
- **refresh_token 过期后必须由账号所有者手动重新走 OAuth 授权流程**，没有自动刷新 refresh_token 的接口
- 工程实现：定时任务监控所有授权账号的 refresh_token 剩余有效期，提前 7 天在管理后台提示运营重新授权；过期窗口内的定时任务降级为"跳过该账号 + 告警"

**核心接口**：
- `VideoAd`：素材上传，期望支持 `UPLOAD_BY_URL`（传 OSS 外链让千川拉取），POC 阶段实测确认
- `UniPromotion.Create`：创建全域推广计划
- `Report.UniPromotionGet`：数据回流（见模块 5）

成片合成后先写入 OSS，投放时千川从 OSS 直接拉取，避免双重上传。后端用 Python（FastAPI），SDK 无官方 Python 版，直接 HTTP 调用 + 参考第三方 Go SDK 的请求体结构。

**ID 链路与关系（已核实）**：
- **1 抖音号 = 1 个全域推广计划**（启用全域即暂停标准推广），无多计划 AB 测试场景
- 1 计划 ↔ N 创意/素材（AB 测试通过同一计划下挂多条创意实现）
- 关系：`成片 (creative) → 千川素材 (creative.material_id) ⊂ 千川计划 (account.campaign_id)`
- `account` 表挂 `campaign_id` 即可，不必每条成片冗余存
- 数据回流粒度（计划层 / 创意层 / 素材层）需 POC 实测，决定 `creative_metrics` 表的最细维度

### 模块 5：数据报表

每日凌晨 Celery Beat 定时任务触发，遍历所有投放中的成片，按千川计划 ID 批量拉取 T+1 数据，写入 PostgreSQL（与主库共用）。展示维度按 PRD 定义：消耗、展示、点击、转化、ROI，外加每日产出数量 + 效果走势趋势图。

**异常兜底**：
- 拉取失败写入重试队列，连续 3 次失败告警
- 提供管理后台 CSV 手动导入，按"千川计划 ID + 日期" upsert
- 未关联计划的成片，UI 上列出供运营手动绑定

**爆款/死款标记**由运营手动操作（`creative.label ∈ {NORMAL, HOT, DEAD}`），MVP 不做自动判定 —— 不同品类和投放周期下"高低"的边界差异太大。

开发预估约 2-3 周。

### 模块 6：素材库维护

三张表的关系模型：

```
material（素材主表）── usage_count 冗余字段，方便排序筛选
material_usage（素材↔成片，多对多）── 成片合成完成时批量写入
creative_metrics（成片投放数据，模块 5 落库）
```

素材看板展示三层聚合：被用在哪些成片、累计使用次数、总消耗、平均 ROI、爆款数量。MVP 阶段走实时 join 查询（PostgreSQL 索引足够），Phase 2 素材上 10K+ 再考虑物化视图。

上传去重 MVP 不做。允许同一文件多次入库（不同标签 / 不同类别的合法场景），运营可手动删除冗余。

开发预估约 1 周。

## 第二部分：整体架构

模块 1-6 封装为一个 Skill（营销内容自动化生产），接入已有 Agent 框架。数据报表独立后端服务，不进 Skill。

**Skill 内部三个人工卡点**：选脚本 / 确认成片 / 选标题。框架是否支持 pause/resume 待确认，决定 Skill 是单段还是按卡点拆成多段。

**技术栈**：FastAPI（Python） + PostgreSQL + Celery + Redis + 阿里云 OSS。部署方案 Docker Compose（MVP）vs K8s，待定。

## 关键风险

| 风险 | 等级 | 缓解策略 |
|------|------|---------|
| 千川 API 未开放或申请周期长 | 极高 | 最优先验证；备选 RPA |
| Seedance 2.0 企业资格 + 保证金门槛 | 高 | 与千川资格申请同步推进；备选其他视频生成模型 |
| 千川 refresh_token 30 天过期需手动重授权 | 中 | 定时监控 + 提前 7 天提示；过期窗口降级 |
| 抖音链接爬取合规问题 | 高 | MVP 只做本地上传，链接解析延后 |
| 分镜拆解准确率不达标 | 高 | 提前 POC，设人工修正兜底 |
| AI 补位画面与实拍风格差异大 | 中 | 以前后帧为参考输入，默认需人工确认 |
| 视频合成性能瓶颈 | 中 | 异步任务队列，评估是否需 GPU |
