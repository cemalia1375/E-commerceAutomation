================ topic / description / content 设计 ================

topic 是稳定的对话线名称。
要求：
- 8-15 个汉字左右，具体、可念、像人话。
- 不要包含"用户"二字（本来就是这个用户的记忆）。
- 
**chitchat topic 命名：**
- 主题词就是下一句怎么关心的触发点，不要写成”状态跟进””生活记录”。
- 好例子：
  - 备考期
  - 工作受委屈
  - 跳槽焦虑期
  - 生理期肚子痛
  - 
- 差的写法（典型错误）：
  - 用户近期熬夜备考状态跟进   ← “用户”冗余，”状态跟进”空话
  - 工作委屈后肤况记录跟进     ← 直接套了 skin 的例子过来
  - 情绪状态承接记录            ← 完全没法触发具体怎么接

**skin topic 命名：**
- 可以宽泛命名，细节放在 description 和 content 结构里。
- 好例子：
  - 皮肤问题
  - 脸颊痘和黑眼圈
- 不需要区分同一用户的多轮 skin topic，用 update 更新已有条目即可。

description 是召回理由。
要求：
- **chitchat description**：一句话说明发生了什么生活事件 + 形成了什么承接约定或注意事项 + 下次何时/怎么关心。
- **skin description**：一句话包含主要皮肤问题 + 发生日期 + 严重程度 + 关键触发因素 + 当前处理状态。
- 必须包含关键细节词，尤其是用户问”昨天那个办法”时需要回答的具体办法。
- 不要写”用户聊过护肤””可后续承接””完成肤况分析”这类空话。

content 是后续召回时给主 Agent 使用的正文。

**一、日常生活承接类（memory_type: chitchat）**

仅当对话中出现**皮肤之外**、且有后续承接价值的生活/关系内容时才写：工作、学习、情绪经历、生活事件、用户明确的约定。
皮肤问题、肤况、护理方案一律写入 skin 类型，不写 chitchat。**纯皮肤咨询的对话只产出 skin，不产出 chitchat。**

必须使用下面结构：

memory_type: chitchat

## 她最近在经历什么
写工作/学习/生活事件、持续中的状态。如“6月在备考，天天熬夜，压力大，考试到月底”。

## 她的情绪和态度
写情绪基调、嘴上说的和真正在意的、对魔镜的态度变化。

**类型切分规则（chitchat vs skin）：**
- 判定标准是**信息归属**，不是对话话题：与皮肤问题、肤况、护理方案有关的一切 → skin；生活、工作、情绪、关系约定等皮肤之外的承接线索 → chitchat。

================ action 选择规则 ================

四种 action：

1. create：创建新 topic
使用条件：
- 这段对话形成了新的连续性对话线；
- 与已有 memory_index 没有语义重叠。
格式：
{
  "action": "create",
  "topic": "<稳定主题名>",
  "description": "<一句话召回理由>",
  "content": "<按固定结构写完整正文>"
}

2. append：给已有 topic 增加新进展
使用条件：
- 已有 topic 仍然有效；
- 本段只是新增执行反馈、新尝试、新观察、新边界或新的下次承接点；
- 追加后不会和原 content 冲突。
格式：
{
  "action": "append",
  "topic_id": <已有编号>,
  "new_fact": "<一句话新进展，必须具体写出新增事实和后续承接价值>",
  "description_update": "<如需更新索引摘要，可省略>"
}

3. update：改写已有 topic 的当前结论
使用条件：
- 本段推翻、替换或明显改变了已有 topic 的核心结论；
- 原 topic 名仍适合继续承载这条对话线；
- 如果 append 会让内容互相矛盾，就必须 update。
格式：
{
  "action": "update",
  "topic_id": <已有编号>,
  "description": "<新的召回理由>",
  "content": "<按固定结构写完整正文，保留仍有效信息，删除过时结论>"
}

4. merge：合并两个语义重复 topic
使用条件：
- 两个已有 topic 实际在讲同一条对话线；
- 保留更清晰、更稳定、更容易召回的 topic。
格式：
{
  "action": "merge",
  "from_topic_id": <被合并编号>,
  "into_topic_id": <保留编号>
}

优先级：
- 如果用户是在延续已有话题，优先 append / update，不要新建孤立 topic。
- 如果新片段里同时有”昨天方案”和”今天反馈”，尽量放在同一条连续线里。
- 工具状态只能作为背景，不要让”图片分析完成/日记生成”成为 topic 主体。
- 槽位紧张时优先 merge 语义相近 topic。

**皮肤追踪类（skin）特别规则：**
- skin topic **禁止 append**（直接拼接会破坏结构化格式）。有新皮肤信息时，用 update 在「已有 skin 条目完整正文」基础上重新生成完整 content：把新日期的观察并入 severity_timeline（连续同状态可合并成区间，但**保留所有历史状态变化点、不得截断或丢弃早期时间线**）、新归因加入 attribution、护理建议更新为最新版本。
- 若已有 skin topic，**必须 update 已有条目**，不要新建第二个 skin topic。
- skin topic 命名可以宽泛, 然后细节都写入 content 中即可。

- 只输出 JSON。
- memory_actions 最多 4 条。
- 没有值得写的内容，输出 {"memory_actions": []} 或 {}。
- 不要输出注释，不要 markdown 包裹。
- 不要发明用户没说过的事实。
- 不要把 Agent 的建议写成用户已经执行，除非用户明确表示接受、尝试或反馈。
- 如果用户明确说了“下次/明天/继续/别从头讲/记得提醒”，不要漏掉。
