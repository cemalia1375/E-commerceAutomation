一次用户请求的完整执行链路
以 Mojing 为例，从用户消息到 SSE 流式响应的全程调用路径（Flowcut 结构一致）
1

用户请求进入 FastAPI

Mojing/api/server.py 接收 POST /chat 请求，解析 tenant_key、message、media 等字段。

POST /chat {"tenant_key": "u_001", "message": "帮我分析一下最近的皮肤状态"}
2

SessionStore 查找 / 冷启动 ReactLoop

Mojing/storage/session_store.py 按 tenant_key 查缓存；首次访问时调用 MainAgent.make_context_builder() 和 make_tool_registry() 组装 simpleclaw 的 ReactLoop。

loop = ReactLoop( llm=VolcengineLLM, # simpleclaw/llm/volcengine.py tool_registry=ToolRegistry, # simpleclaw/tools/registry.py context_builder=ContextBuilder # simpleclaw/context/builder.py )
3

ContextBuilder 组装 System Prompt

simpleclaw/context/builder.py 合并：
• 稳定前缀：从 workspace/ 加载的 Agent.md + SOUL.md + TOOL.md（走前缀缓存）
• 动态尾部：业务 Provider 每轮实时计算
• Attention Packets：注入到消息流特定位置

system = [stable_prefix] # ~3200 tokens，前缀缓存 + [dynamic_tail] # 每轮实时计算 messages = history + attention_packets
4

ReactLoop 调用 LLM，流式输出 Token

simpleclaw/core/loop.py 调用 VolcengineLLM.stream_with_retry()，TextChunk 实时 yield 给前端（SSE），同时收集 ToolCallChunk。

async for chunk in llm.stream_with_retry(messages): if isinstance(chunk, TextChunk): yield TextEvent(chunk.token) # → 前端实时看到 elif isinstance(chunk, ToolCallChunk): tool_calls.append(...) # → 收集待执行工具
5

工具执行：耦合 vs 解耦

LLM 返回工具调用后，ReactLoop 按 needs_followup 分流：
• coupled（needs_followup=True）：并行执行，结果注入历史 → 再次调用 LLM
• decoupled（needs_followup=False）：asyncio.create_task() 后台触发，不阻塞 LLM

coupled = [SearchMaterials] # 等待结果 → 注入 LLM decoupled = [DecomposeVideo] # fire & forget
6

durable 工具 → 任务队列 → Worker 消费

长运行工具（execution_mode="durable"）调用 RuntimeServices.submit_task() 入队；TaskWorker 后台消费，完成后写 DB。

TaskQueue → stream: VIDEO_DECOMPOSE TaskWorker → executor: make_decompose_video_executor()
7

Turn 结束 → Postprocess / 记忆提取

每轮结束后，PostprocessHook 和 ContextCompressor 被触发；超出 token 窗口的历史消息被压缩，consolidated_from 指针前移。

compressor.maybe_compress(active_messages) postprocess_hook.on_turn_end(session_id, messages)
真实数据流追踪
以「用户上传视频请求分解脚本」为例（Flowcut），追踪数据在各层的转换

示例场景：用户上传抖音视频，请 Flowcut 分解素材并生成广告脚本

输入: POST /chat
  {
    "tenant_key": "fc_u_001",
    "message": "帮我分析这个视频，提取素材并生成3个脚本",
    "media": ["https://oss.example.com/videos/v_001.mp4"]
  }

Step 1 → SessionStore 冷启动

Flowcut/storage/session_store.py
↓ sessions["fc_u_001"] → cache miss → 冷启动
↓ main_agent.make_context_builder("fc_u_001", stage="default") # 无三阶段
↓ main_agent.make_tool_registry("fc_u_001") # 注册 6 个工具
→ ReactLoop 实例化完成（端口 8001）

Step 2 → ContextBuilder 拼接 System Prompt

simpleclaw/context/builder.py
↓ stable_sections = [Agent.md, SOUL.md, TOOL.md, compliance.md]（~2800 tokens，走前缀缓存）
↓ TaskContextProvider → 当前进行中任务 0 个（stub 返回 []）
→ messages: [system(稳定), system(动态), user_with_video_url]

Step 3 → LLM 决策调用 decompose_video

simpleclaw/core/loop.py + simpleclaw/llm/volcengine.py
↓ TextChunk → SSE: "好的，我来帮你分解这个视频..."
↓ ToolCallChunk → {
    id: "call_001", name: "decompose_video",
    arguments: {video_url: "https://oss.example.com/videos/v_001.mp4"}
  }

Step 4 → DecomposeVideoTool 入队（durable）

Flowcut/tools/decompose_video.py → simpleclaw/runtime/services.py
↓ execution_mode = "durable"
↓ prepare_task() → TaskEnvelope {
    task_id: "task_abc", stream: "VIDEO_DECOMPOSE",
    payload: {video_url, tenant_key}
  }
↓ RuntimeServices.submit_task() → 写入 Redis/InMemory Stream
↓ ToolResult { ok: true, task_id: "task_abc" } → 注入 LLM
→ LLM 回复「已提交分解任务，完成后可生成脚本」→ DoneEvent

Step 5 → Worker 后台处理（当前 NotImplementedError 占位）

Flowcut/runtime/executors.py → make_decompose_video_executor()
↓ TaskWorker 消费 VIDEO_DECOMPOSE stream
↓ raise NotImplementedError # ASR + Gemini 场景分解待实现
↓ 预期：material_repo.create(status=PROCESSING) → ASR → 分割 → status=READY
→ 下次对话 TaskContextProvider 注入任务状态摘要
架构关系：业务模块 × simpleclaw
Mojing 和 Flowcut 均以相同方式接入 simpleclaw，体现框架的可复用性
Mojing × simpleclaw — 5 个连接点

连接点 ① Tool 继承

Mojing 的所有工具都继承自 simpleclaw.tools.base.Tool。ReactLoop 只认 Tool 抽象，不需要知道 Mojing 的存在。

Mojing/tools/image_tools.py → class AnalyzeImageTool(Tool)
Mojing/tools/deep_research.py → class DeepResearchTool(Tool)

连接点 ② ContextBuilder 注入

MainAgent.make_context_builder() 实例化 ContextBuilder，并把 Mojing 的 Provider 实现注入进去。按三阶段（novice/explore/mature）动态组装。

连接点 ③ ToolRegistry 组装

MainAgent.make_tool_registry() 实例化 ToolRegistry，按用户阶段（staged_tool_factories）动态决定注册哪些工具。

连接点 ④ RuntimeServices 共享

单一 RuntimeServices 实例被 ToolRegistry 和所有 TaskWorker 共享，形成统一任务调度中心。9 个 Worker 并行运行。

连接点 ⑤ SubagentBase 继承（Mojing 独有）

SkinDiarySubagent 和 DeepReportSubagent 继承 SubagentBase，内部各运行一个独立 ReactLoop。Flowcut 无此层。

Flowcut × simpleclaw — 4 个连接点（无 SubagentBase）

连接点 ① Tool 继承（6 个工具）

6 个 Flowcut 工具全部继承 simpleclaw.tools.base.Tool。核心逻辑以 raise NotImplementedError 占位，框架接入已完成。

decompose_video / generate_scripts / search_materials
compose_video / check_task_status / publish_to_qianchuan

连接点 ② ContextBuilder 注入（单一 default 阶段）

Flowcut/agent/main_agent.py 注入 TaskContextProvider（当前 stub）。无三阶段切换，所有用户使用同一工具集和提示词。

连接点 ③ ToolRegistry 组装（无 staged_tool_factories）

所有 6 个工具在 build_container() 时通过 lambda 工厂注册，无按阶段动态启停逻辑。

连接点 ④ RuntimeServices 共享（5 个 Stream）

5 个 TaskStream：VIDEO_DECOMPOSE / SCRIPT_GENERATE / MATERIAL_SEARCH / VIDEO_COMPOSE / QIANCHUAN_PUBLISH。5 个 Worker 并行，Executor 全部占位。

架构依赖方向（单向，两个业务模块均遵守）
Mojing → imports → simpleclaw
Flowcut → imports → simpleclaw
simpleclaw ✗ 永远不导入 ✗ Mojing / Flowcut
Mojing ✗ 不导入 ✗ Flowcut（业务模块互相独立）

simpleclaw 提供协议 / 抽象 — Tool, LLMProvider, Memory, SubagentBase...
Mojing 实现美妆垂直领域业务逻辑（9 Worker，子 Agent，三阶段旅程）
Flowcut 实现千川内容生产业务逻辑（5 Worker，无子 Agent，单阶段）
